KB Vault Resolution Bug Report

## Bug: Project vault incorrectly overrides Personal vault

**Issue**: The KB extension's vault resolution logic always selects the **project** vault when running from `~/.pi/agent` directory, even when the user has a personal KB vault in `~/.kb/` and wants to use that.

**Reproduction Steps:**
1. User has a personal KB vault at `~/.kb/` with mode "personal"
2. User has a project KB vault at `~/.pi/agent/.kb/` with mode "project"  
3. User runs pi from `/home/debasmitr/.pi/agent`
4. Expected: Use personal KB (`~/.kb/`) - shows "🧠 KB — personal"
5. Actual: Uses project KB (`~/.pi/agent/.kb/`) - shows "🧠 KB — project"

**Code Location:**
`/home/debasmitr/.pi/agent/extensions/kb/lib/vault.ts`, `resolveVaultContext()` function

**Root Cause:**
The vault resolution walks up from the current directory and finds `~/.pi/agent/.kb/config.json` before reaching `~/.kb/config.json`. The current logic treats the first `.kb/` found as a project vault, ignoring the user's explicit personal vault location.

**Expected Behavior:**
- If user has `~/.kb/` with explicit config, that should be used for personal mode
- Personal vault in home directory should take precedence over nested project vaults in `.pi/agent/`

**Current Behavior:**
- Nested `.kb/` anywhere in path walk is treated as project mode
- User's personal KB at `~/.kb/` is ignored when running from `.pi/agent/`

**Impact:**
- Users cannot use their personal KB while working within `.pi/agent/`
- Incorrect mode indicator shows "project" instead of "personal"
- User's personal knowledge base contents are inaccessible
