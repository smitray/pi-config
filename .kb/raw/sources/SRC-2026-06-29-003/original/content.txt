# Agent Sandbox Options for Pi

## Gondolin (Recommended)
- Same org as Pi (earendil-works)
- Micro-VM (QEMU by default, optional libkrun backend)
- Sub-second startup, ~200MB guest image (cached locally)
- Programmable HTTP/TLS egress with allowlists + request/response hooks
- Secret injection via placeholders — guest sees fake token, host injects real one only for allowed hosts
- Programmable VFS mounts in JavaScript
- Already has a Pi extension that routes read/write/edit/bash into VM, mounts cwd at /workspace
- 1.5k stars, 386 commits, Apache-2.0
- CLI: `npx @earendil-works/gondolin bash`
- Requirements: Node.js >= 23.6.0, QEMU (apt/brew)

## Podman/Docker
- Container-based (OCI), namespace isolation (shared kernel)
- Broader ecosystem, familiar tooling
- Simple Dockerfile + bind mount workflow
- API keys enter the container (less secure than placeholder injection)
- Heavier startup (~1-2s)

## NVIDIA OpenShell
- Policy-controlled managed sandbox
- Local gateway (Docker/Podman/VM) or remote Kubernetes gateway
- Filesystem, process, network, credential, and inference controls
- Gateway injects credentials — code calls https://inference.local
- No host filesystem access in remote mode (upload/download required)
- Team/enterprise scale

## Comparison Table
| | Gondolin | Podman/Docker | OpenShell |
|---|---|---|---|
| Type | micro-VM | Container | Managed sandbox |
| Startup | Sub-second | ~1-2s | Varies |
| Isolation | Kernel-level | Namespace | Policy-controlled |
| Network | JS hooks (allowlist) | CNI rules | Gateway policy |
| Secrets | Placeholder injection | Keys enter container | Gateway injection |
| Pi integration | ✅ Built-in extension | Manual Dockerfile | CLI sandbox create |
| Overhead | ~200MB cached | Image layers | Gateway + VM |

## Decision: Gondolin
Same org as Pi, has a ready extension, fastest startup, best secret handling pattern.
